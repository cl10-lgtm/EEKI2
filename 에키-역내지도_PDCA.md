## P — Plan (기획 목표 및 설계 원칙)

### 기능 목표

1. **경쟁사가 끝나는 지점에서 시작** — 경로 검색 완료 후 "개찰구 통과 ~ 플랫폼 도착"까지 앱이 안내
2. **정보 과다 역설 해결** — 안내판이 많은 대형 역일수록 필요한 정보 1개만 표시
3. **오프라인 완전 작동** — 지하 LTE 없이 MVP 5개 역 전체 턴바이턴 제공

> U6: *"역 내가 커서 수많은 안내 위치와 포스터로 인해(정보 과다) 헷갈림"*
> U8: *"신주쿠 역같은 경우 다수의 출구로 인해 위치 찾기가 어려움"*
> U9: *"탑승구 못 찾음"*
> U10: *"코인락커(물품보관함) 위치 찾기가 어려웠음"*

### 설계 원칙

- **한 번에 하나** — 현재 해야 할 행동 1개만 크게 표시, 전체 경로는 접어서 숨김
- **랜드마크 기준 안내** — "7번 기둥", "초록색 자동발매기 옆" 등 실제 보이는 것 기준
- **오프라인 선행** — 모든 역 내 데이터는 사전 다운로드, 네트워크 의존 없음
- **경로 검색과 연속성** — 경로 결과의 "탑승 플랫폼" 정보가 역내 지도로 자동 연결

### FEAT-02. 역내 지도 진입 시나리오

**시나리오 A — 경로 검색 결과에서 자동 연결 (메인)**

```
경로 결과 화면:
  하카타 → 텐진   8분   ¥210
  🚇 공항선   1번 플랫폼 탑승
                    ↓
  [ 역내 길안내 시작 ]  ← 탭 시 FEAT-03 진입
  (하카타역 → 1번 플랫폼 자동 목적지 설정)
```

**시나리오 B — 직접 진입 (시설 찾기)**

```
홈 탭바 > [🗺 지도] > 역 선택 > 목적지 선택

목적지 옵션:
  ┌────────────────────────────┐
  │  무엇을 찾으시나요?         │
  │  [🚇 플랫폼]  [🚪 출구]    │
  │  [🚻 화장실]  [🔒 코인락커]│
  │  [ℹ️ 안내소]  [🏧 ATM]    │
  │  [🛗 엘리베이터]           │
  └────────────────────────────┘
```

**시나리오 C — 잔액 부족 ATM에서 연결**

```
에키-결제흐름 FEAT-03 ATM 지도
  → "ATM까지 길안내" 탭
  → 역내 지도 자동 진입 + ATM 목적지 설정
```

### FEAT-03. 턴바이턴 단계별 안내

**트리거**: FEAT-02 시나리오 중 하나로 목적지 설정 완료

**경로 계산 알고리즘**
```
입력: origin_node (현재 위치 or 개찰구), destination_node
알고리즘: Dijkstra (가중치: walk_seconds)
우선순위: 엘리베이터 우선 옵션 (설정에서 on/off)
출력: 노드 배열 + 각 엣지 instruction_ko
```

**현재 위치 결정 방식**
```
1순위: 경로 검색 기반 — 어느 개찰구로 진입하는지 추론
  예) "하카타역 도착 → 공항선 1번 플랫폼"
    → 자동으로 "공항 방면 개찰구"를 origin으로 설정

2순위: 유저 선택 — "지금 어디 계세요?"
  [개찰구 통과 직후] [이미 역 내부에 있어요]
  → 가장 가까운 주요 노드 선택 UI

3순위: 랜드마크 선택 — "주변에 보이는 것"
  [초록색 자동발매기] [편의점] [엘리베이터] ...
  → 선택한 랜드마크 기준 경로 재계산
```

**안내 화면 구조**
```
┌──────────────────────────────────────┐
│  하카타역 → 1번 플랫폼   3분         │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  │
│                                      │
│  ┌──────────────────────────────┐   │
│  │                              │   │
│  │   ↑  직진으로 40m            │   │  ← 현재 단계 (크게)
│  │                              │   │
│  │   편의점 지나서               │   │  ← 랜드마크
│  └──────────────────────────────┘   │
│                                      │
│  다음: 계단 내려가서 우회전          │  ← 다음 단계 (작게)
│  ──────────────────────────────────  │
│  전체 경로 보기 ∨                    │  ← 펼치면 전체 스텝 목록
│  ──────────────────────────────────  │
│  [🚻 화장실]  [🔒 락커]  [길 잃었어요]│  ← 빠른 시설 접근
└──────────────────────────────────────┘
```

**전체 경로 스텝 목록 (펼쳤을 때)**
```
✅ 1. 남쪽 개찰구 통과 직후 직진
▶ 2. 편의점 지나서 40m 직진       ← 현재
   3. 계단 내려가기 (지하 2층)
   4. 우회전 후 20m
   5. 1번 플랫폼 도착
```

**층 변경 처리**
```
계단·에스컬레이터·엘리베이터 노드 도달 시:
  → 진동 햅틱 1회
  → 화면: "지하 2층으로 내려가세요"
  → 아이콘: 🔽 계단 / 🔽 에스컬레이터 / 🛗 엘리베이터
  → 다음 단계 자동 대기 (탭 없이 스크롤)
```

**"길 잃었어요" 긴급 리셋**
```
탭 시:
  → "지금 어디 계세요?" 랜드마크 선택 UI 재표시
  → 선택 후 현재 위치부터 경로 재계산
  → 역무원 카드 진입 옵션 함께 표시
```

### FEAT-05. 오프라인 데이터 관리

**번들 전략**
```
앱 설치 시 기본 내장:
  - MVP 5개 역 경량 데이터 (출구·플랫폼 번호만, 각 ~200KB)

온보딩 or 설정에서 다운로드:
  - 풀 데이터 (턴바이턴 + 시설 + 랜드마크, 각 ~2MB)
  - 목적지 도시 선택 시 해당 역 자동 추천
```

**버전 관리**
```json
{
  "station_cache": {
    "shinjuku": {
      "version": "2026.04",
      "downloaded_at": "2026-04-20T09:00:00Z",
      "expires_at": "2026-05-20T09:00:00Z",
      "data_type": "full"
    }
  }
}
```

**갱신 감지 API**
```
GET /api/v1/maps/versions
Response:
  {
    "shinjuku": { "latest": "2026.04", "changelog": "남쪽 출구 A1 폐쇄" },
    "hakata":   { "latest": "2026.03", "changelog": null }
  }

갱신 시: 홈 배너 "신주쿠역 지도 업데이트 있음 → 다운로드"
오프라인 시: 기존 캐시 사용 + "저장된 지도 기준" 배지 표시
```

## C — Check (검증 지표)

### 정량 지표

| 지표 | 목표값 | 측정 이벤트 |
|------|--------|------------|
| 역내 길안내 완료율 | ≥ 65% | `indoor_nav_started` → `destination_reached` |
| 평균 안내 스텝 수 (신주쿠 기준) | ≤ 6스텝 | 경로 계산 결과 node 수 |
| "길 잃었어요" 탭 비율 | ≤ 15% | `lost_button_tapped` / `indoor_nav_started` |
| 출구 핀포인트 클릭률 | ≥ 50% | `exit_recommend_tapped` |
| 오프라인 상태 안내 성공률 | 100% | `indoor_nav_offline_success` |
| 풀 데이터 다운로드율 (5개 역) | ≥ 35% | `map_full_downloaded` |

### 정성 지표

- "앱이 알려주는 대로 따라갔을 때 실제로 도착했나요?"
- "안내 중 헷갈렸던 단계가 있었나요? 어느 부분이었나요?"
- "랜드마크 설명이 실제 역 안에서 보이는 것과 맞았나요?"

## A — Act (개선 트리거 및 로드맵)

### 즉시 개선 트리거

| 상황 | 기준 | 액션 |
|------|------|------|
| 길안내 완료율 저조 | < 45% | "길 잃었어요" 탭 발생 위치 분석 → 해당 구간 랜드마크 재작성 |
| 특정 스텝 이탈 집중 | 특정 노드 이탈 > 30% | 해당 구간 instruction_ko + 랜드마크 현장 재검증 |
| "길 잃었어요" 비율 높음 | > 25% | 현재 위치 재설정 UI 개선, 랜드마크 선택지 확대 |
| 오프라인 캐시 만료 빈번 | 만료 후 사용 > 20% | 캐시 유효기간 30일 → 60일 연장 검토 |

### 다음 단계 로드맵

```
MVP (v1.0)  ── 5개 역, 턴바이턴 핵심
  ├── FEAT-01  5개 역 데이터 구조 + 오프라인 번들 (경량)
  ├── FEAT-02  시나리오 A (경로 결과 자동 연결) + B (직접 진입)
  ├── FEAT-03  턴바이턴 안내 (출구·플랫폼·화장실·ATM)
  └── FEAT-04  출구 핀포인트 추천

v1.1  ── 정밀화 + 개인화
  ├── FEAT-05  풀 데이터 다운로드 + 버전 관리
  ├── FEAT-02  시나리오 C (ATM 지도 연동)
  └── 엘리베이터 우선 경로 옵션 (접근성)

v1.2  ── 데이터 확장
  ├── 상위 5개 역 → 10개 역 확장
  ├── 유저 제보 기반 랜드마크 보정
  └── 역 구조 변경 자동 감지 (변경 이력 알림)
```
