## P — Plan (기획 목표 및 설계 원칙)

### 기능 목표

역무원 카드는 **번역 앱의 대체재가 아니라 심리적 부담의 제거제**다.
유저가 타이핑하고 기다리는 시간 동안 쌓이는 눈치·압박·줄 부담을 없애는 것이 핵심.
카드를 역무원에게 보여주는 행위 자체가 "내가 도움이 필요한 외국인"임을 명확히 전달해,
역무원도 즉시 상황을 파악하고 대응할 수 있게 한다.

> U6: *"오래 번역 시에 눈치 보이고, 줄에 대한 부담감이 생김"*
> U6: *"역무원과의 번역 의지 부족으로 의사소통이 안돼서 불편함"*
> U7: *"심리적 압박이 커서 길치 현상이 심화됨"*

### 설계 원칙

- **3초 접근** — 홈 하단 버튼 → 상황 선택 → 카드 표시까지 3탭 이내
- **텍스트 최소화** — 역무원이 한눈에 읽을 수 있도록 일본어만 크게, 한국어 설명은 작게
- **발음 병기** — 히라가나 + 로마자로 유저도 직접 읽을 수 있게 (자신감 부여)
- **오프라인 완전 작동** — 지하 LTE 없이도 모든 카드 즉시 표시
- **상황 확장 가능 구조** — 카드 DB를 서버에서 관리해 앱 업데이트 없이 추가 가능

### FEAT-01. 홈 하단 고정 빠른 접근 버튼

**트리거**: 앱 실행 후 홈 화면 (온보딩 완료 여부 무관하게 항상 노출)

**목적**: 역무원 카드가 필요한 순간은 예측 불가 — 언제든 1탭으로 진입 가능해야 함

**UI 구조**
```
홈 화면 하단 고정 탭바:

  [🔍 경로]   [🗺 지도]   [🗣 대화카드]   [⚙ 설정]
                              ↑
                     항상 노출, 탭바 중앙 배치
                     (긴급 상황에서 엄지 접근 최적화)
```

**상태별 버튼 동작**
```
일반 상태     → 탭 클릭 시 FEAT-02 (상황 선택 화면) 진입
경로 진행 중  → 탭 클릭 시 현재 구간 관련 상황 카드 자동 추천 후 FEAT-02
오프라인      → 동일 작동 (전체 로컬 저장)
```

### FEAT-02. 상황 선택 화면

**트리거**: 홈 하단 "대화카드" 버튼 탭

**카테고리 및 상황 목록 (전체)**

```
📋 카테고리 1: 개찰구 문제
  ├── 잔액 부족으로 개찰구가 열리지 않아요
  ├── 개찰구에 카드가 걸렸어요
  ├── 표를 잃어버렸어요
  └── 개찰구 오류가 떴어요

📋 카테고리 2: 열차 탑승 실수
  ├── 열차를 잘못 탔어요
  ├── 정류장을 지나쳤어요
  ├── 환승 방법을 모르겠어요
  └── 이 열차가 ○○역에 서나요? (목적지 입력)

📋 카테고리 3: 역사 내 위치
  ├── 화장실이 어디에 있나요?
  ├── 코인락커(짐 보관함)가 어디에 있나요?
  ├── 안내소가 어디에 있나요?
  ├── ○번 출구가 어디에 있나요? (번호 입력)
  └── 엘리베이터가 어디에 있나요?

📋 카테고리 4: 분실·긴급
  ├── 지하철 안에 물건을 두고 내렸어요
  ├── 지갑을 잃어버렸어요
  ├── 몸이 좋지 않아요 (긴급)
  └── 경찰서(파출소)가 어디에 있나요?

📋 카테고리 5: 결제·환불
  ├── 환불을 받고 싶어요
  ├── 요금이 잘못 청구된 것 같아요
  └── IC카드를 반납하고 싶어요
```

**화면 구조**
```
┌──────────────────────────────────────┐
│  ← 대화 카드                          │
│  [개찰구] [열차] [위치] [분실] [결제] │  ← 카테고리 탭
│  ──────────────────────────────────  │
│                                      │
│  💳 개찰구 문제                       │
│  ┌──────────────────────────────┐   │
│  │ 잔액 부족으로 개찰구가 안 열림 │   │  ← 탭하면 FEAT-03
│  └──────────────────────────────┘   │
│  ┌──────────────────────────────┐   │
│  │ 개찰구에 카드가 걸렸어요      │   │
│  └──────────────────────────────┘   │
│  ...                                 │
└──────────────────────────────────────┘
```

**검색 기능**
```
상단 검색바: 한국어 키워드 입력 → 관련 카드 실시간 필터
예) "화장실" 입력 → "화장실이 어디에 있나요?" 즉시 노출
```

### FEAT-03. 대화 카드 표시 화면

**트리거**: FEAT-02에서 상황 선택

**카드 데이터 스키마**
```json
{
  "card_id": "gate_no_balance",
  "category": "gate",
  "situation_ko": "잔액 부족으로 개찰구가 열리지 않아요",
  "display_ja_kanji": "ICカードの残高が不足しています。\nチャージをお願いしたいのですが、\nどうすればよいですか？",
  "display_ja_hiragana": "あいしーかーどの ざんだかが ふそくして います。\nちゃーじを おねがいしたいのですが、\nどうすれば よいですか？",
  "display_romaji": "Aishī kādo no zandaka ga fusoku shite imasu.\nChāji wo onegai shitai no desu ga,\ndō sureba yoi desu ka?",
  "display_en": "My IC card has insufficient balance.\nCould you tell me how to charge it?",
```

**화면 접근성 처리**
```
- 화면 자동 밝기 최대로 올리기 (역무원이 잘 볼 수 있도록)
- 가로 모드 지원 (더 큰 글씨)
- 폰트 크기 설정과 무관하게 카드 텍스트 고정 크기
- 화면 꺼짐 방지 (카드 표시 중 슬립 모드 비활성화)
```

**변수 포함 카드 처리** (목적지·출구번호 입력 필요)
```
카드 ID: "train_does_stop"
template_ja: "{station_ja}に止まりますか？"

→ 카드 선택 시 입력 모달:
  "목적지 역 이름을 입력하세요"
  [ 자동완성 검색 ]
  
→ 입력 후 카드에 반영:
  "新宿に止まりますか？"
  "しんじゅくに とまりますか？"
  "Shinjuku ni tomarimasu ka?"
```

### FEAT-04. 카드 DB 서버 관리

**목적**: 앱 업데이트 없이 새 상황 카드 추가·수정 가능

**API 명세**
```
GET /api/v1/cards?lang=ko&version={local_version}
Response:
  {
    "version": "2026.04.24",
    "updated": true,
    "cards": [ ...card_schema 배열 ]
  }

업데이트 전략:
  - 앱 실행 시 백그라운드 조용히 동기화
  - 버전 동일 시 → 다운로드 스킵
  - 오프라인 시 → 로컬 캐시 사용
  - 최초 설치 시 → 번들 기본 카드 30종 내장 (오프라인 보장)
```

**번들 내장 카드 (최초 30종)**
```
개찰구 문제  ×4
열차 탑승    ×4
역사 위치    ×5
분실·긴급    ×4
결제·환불    ×3
기본 인사    ×10  (감사합니다, 죄송합니다, 잠깐만요 등)
```

## C — Check (검증 지표)

### 정량 지표

| 지표 | 목표값 | 측정 이벤트 |
|------|--------|------------|
| 카드 접근까지 소요 탭 수 | ≤ 3탭 | `fab_tap` → `card_displayed` |
| 카드 표시 후 체류 시간 | ≥ 10초 | `card_displayed` → `card_closed` |
| 카드 실사용률 (표시 후 5초 이상) | ≥ 40% | 체류 10초 이상 비율 |
| 즐겨찾기 등록률 | ≥ 20% | `card_favorited` / `card_displayed` |
| 연관 기능 딥링크 클릭률 | ≥ 15% | `related_feature_tapped` |
| 오프라인 상태 카드 표시 성공률 | 100% | `card_displayed_offline` |

### 정성 지표 (출시 후 현장 관찰)

- "실제로 문제가 해결됐나요? 어떤 카드가 가장 유용했나요?"

## A — Act (개선 트리거 및 로드맵)

### 즉시 개선 트리거

| 상황 | 기준 | 액션 |
|------|------|------|
| 카드 체류 시간 짧음 | 중앙값 < 5초 | 카드 글씨 크기 확대, 화면 밝기 자동 최대 적용 여부 확인 |
| 특정 카드 사용 집중 | 상위 3개 카드 > 70% | 해당 카드 홈 화면 바로가기 추가, 나머지 카드 재검토 |
| 연관 기능 딥링크 무시 | 클릭률 < 5% | 딥링크 위치·문구 변경 또는 제거 후 카드 집중 |
| 오프라인 카드 실패 발생 | 실패율 > 0% | 번들 카드 업데이트 주기 및 캐시 전략 재검토 |
| 즐겨찾기 미사용 | 등록률 < 5% | 즐겨찾기 기능 제거 후 사용 이력 자동 정렬로 단순화 |

### 다음 단계 로드맵

```

  ├── FEAT-04  카드 DB 서버 동기화 (앱 업데이트 없이 카드 추가)

  ├── 경로 연동 자동 추천 (현재 구간 기반 카드 제안)
  └── 역무원 피드백 수집 ("이 카드가 통했나요?" 원탭 리포트)
```

## 미해결 리스크

| 항목 | 내용 |
|------|------|
| 역무원 실제 반응 | 카드가 통하지 않을 경우 신뢰 손상 치명적 → 일본 현지 역무원 대상 사전 테스트 필수 |
| 번역 정확도 | 일본어 경어·문맥 오류 시 오히려 역효과 → 일본어 원어민 감수 필수 (출시 전) |
| 화면 밝기 강제 설정 | iOS/Android 시스템 밝기 제어 권한 정책 확인 필요 |
| 카드 남용 방지 | 카드를 이용한 부당 요구(환불 악용 등) 케이스 설계 검토 필요 |
| 긴급 상황 카드 책임 | "몸이 좋지 않아요" 같은 긴급 카드는 법적 책임 범위 검토 필요 |
